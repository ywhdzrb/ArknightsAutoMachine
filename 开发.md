**Arknights Auto Machine (AAM) 开发计划书**

---

## 1. 项目概要

**产品定位**: 基于AMA分层架构的明日方舟自动化代理工具，通过ADB与模拟器交互，采用标准视觉语言模型(VLM)实现高层决策。

**核心创新**: 首创性将"联想运动架构"(AMA)应用于游戏自动化领域，通过L0-L5分层解决传统端到端AGI的高延迟问题。

**运行环境**: Windows/Linux + Android模拟器(夜神/雷电/蓝叠等)

---

## 2. AMA架构实现映射

| 层级 | 模块命名 | 核心职责 | 关键指标 |
|------|----------|----------|----------|
| **L5** | 决策层(Brain) | 视觉理解、战术规划、干员部署决策 | 决策延迟<2s，API调用频率<0.5次/秒 |
| **L4** | 记忆层(Memory) | 对局状态机、关卡进度存档、操作历史回溯 | 状态切换延迟<50ms，支持断点续打 |
| **L3** | 预测层(Predictor) | 敌人轨迹预测、费用曲线预判 | 预测准确率>85%，纯本地计算<16ms |
| **L2** | 映射层(Mapper) | 语义坐标(第3格/向右)转屏幕像素(650,650) | 映射误差<5px，支持多分辨率适配 |
| **L1** | 反射层(Reflex) | 紧急情况处理(血线危急自动撤退/开技能) | 响应延迟<100ms，独立于L5运行 |
| **L0** | 感知/执行层 | 截图输入(ADB screencap)、触控输出(ADB input) | 截图帧率>15fps，触控精度±3px |

---

## 3. 开发路线图

### 第一阶段：感知执行层 (M1: Foundation)
**周期**: 第1-2周  
**目标**: 建立稳定的ADB通信链路，实现基础感知与执行能力

**交付要求**:
- [ ] ADB设备发现与连接管理器，支持多设备切换
- [ ] 截图模块：稳定获取1920x1080游戏画面，帧率≥15fps，内存占用<200MB
- [ ] 执行模块：封装tap/swipe/drag操作，支持拖拽干员到指定屏幕坐标
- [ ] 基础GUI框架：设备连接面板、实时画面预览窗口(降采样显示)、操作日志面板
- [ ] 手动控制模式：支持点击GUI按钮触发单次ADB操作，用于调试

**验收标准**:
- 连续截图1000帧无崩溃，单帧获取延迟<66ms
- 拖拽操作定位误差<5像素，支持200ms-500ms时长调节
- GUI响应流畅，预览窗口刷新率≥10fps

---

### 第二阶段：空间映射层 (M2: Coordinate System)
**周期**: 第3-4周  
**目标**: 建立游戏内语义空间到屏幕像素的映射数据库

**交付要求**:
- [ ] 关卡配置系统：JSON格式定义网格系统(9x5/11x6等)、UI元素位置、干员栏坐标
- [ ] 坐标映射引擎：支持"列-行"网格坐标转屏幕像素，支持不同分辨率(1080p/2K/4K)自适应
- [ ] 可视化标定工具：GUI辅助工具，允许用户在预览画面上点击标定网格原点和单元格尺寸，自动生成配置文件
- [ ] 初始关卡包：完成5张高频关卡(1-7, CE-6, LS-6等)的精确映射配置

**验收标准**:
- 映射精度：目标像素与实际点击位置偏差<5px
- 配置生成器：新关卡标定时间<3分钟
- 支持分辨率切换时自动重映射，无需重启程序

---

### 第三阶段：认知与记忆层 (M3: Intelligence)
**周期**: 第5-6周  
**目标**: 接入视觉语言模型，实现策略决策与状态管理

**交付要求**:
- [ ] L5决策模块：支持至少一种VLM API(Qwen-VL/GPT-4V/Claude-3V)，输入游戏画面输出结构化JSON指令
- [ ] L4状态机：实现完整的关卡生命周期状态管理(准备→作战→结算)，自动识别胜利/失败画面
- [ ] 记忆系统：SQLite数据库记录每场对局的干员部署序列、费用曲线、操作时间戳
- [ ] 策略配置面板：GUI设置VLM API密钥、模型选择、决策频率控制(保守/激进模式)

**验收标准**:
- VLM识别准确率：干员栏位识别>90%，敌人位置标记>85%
- 状态机：从进入关卡到自动点击"再次作战"的全流程无人值守
- 单局决策延迟(L5推理+L4状态更新)<3秒

---

### 第四阶段：预测与反射层 (M4: Real-time Optimization)
**周期**: 第7-8周  
**目标**: 降低LLM调用频率，提升紧急响应能力，达到实用级实时性能

**交付要求**:
- [ ] L3运动预测：基于OpenCV光流/模板匹配实现敌人轨迹预测，支持1-3秒前瞻预测
- [ ] L1反射弧：独立守护线程监控干员血线，自动触发技能/撤退，响应时间<100ms
- [ ] 分层协调机制：L5负责战略部署(每5-10秒决策一次)，L3/L1负责战术微调(每秒30次本地计算)
- [ ] 性能监控面板：实时显示各层延迟(L0截图耗时、L5推理耗时、总循环周期)

**验收标准**:
- L3预测命中率>80%，将L5调用频率降低50%以上
- 紧急情况(干员濒死)响应时间<150ms，自动开技能成功率>95%
- 系统总延迟(画面变化到执行动作)<500ms(不含L5战略决策)

---

### 第五阶段：工程化与扩展 (M5: Production)
**周期**: 第9-10周  
**目标**: 系统稳定性优化，关卡覆盖扩展，用户体验完善

**交付要求**:
- [ ] 关卡自动识别：无需手动选择配置，通过画面特征自动匹配对应地图配置
- [ ] 异常恢复机制：掉线检测、闪退自动重连、作战失败自动重试策略
- [ ] 日志与回放：完整记录每局操作序列，支持"录像回放"功能用于复盘
- [ ] 安装包构建：PyInstaller打包为独立可执行文件，零依赖部署

**验收标准**:
- 连续运行8小时无内存泄漏，崩溃率<1%
- 支持至少20张不同地图的自动识别
- 非技术用户可在5分钟内完成安装并运行

---

## 4. 技术限制与风险

**硬性约束**:
- **平台限制**: 仅支持Android模拟器(ADB方案)，不支持iOS设备或PC客户端
- **分辨率依赖**: 每新增一种屏幕分辨率(如平板比例)，需重新标定L2映射层
- **成本约束**: L5层依赖第三方VLM API，高频调用产生费用，必须通过L3/L1层缓存优化控制成本

**合规风险**:
- 操作行为需加入随机延迟(50-200ms抖动)，避免被识别为机械脚本
- 建议仅用于单机刷图，避免在联机/合约排行榜场景使用
